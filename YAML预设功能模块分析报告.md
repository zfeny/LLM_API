## 核心功能模块

### 1. 吐槽/评论/弹幕机制 ⭐⭐⭐⭐⭐

#### 功能描述
一种创新的**注意力分配机制**，通过让AI以"作者身份"对创作过程进行实时评论，实现动态的思维链功能。

#### 实现方式（Izumi预设）

```yaml
# 吐槽式思维链
<tucao>
【首条吐槽：构思与规划】
- 泉此方对当前创作任务的理解
- 需要注意的重点
- 文风选择
- 结构规划
</tucao>

# 正文创作...

<tucao>
【中间吐槽：过程监督】
- 检查是否遵守了规则
- 调整节奏
- 纠正偏差
</tucao>

# 继续正文...

<tucao>
【末条吐槽：总结与展望】
- 本次创作的完成度
- 遗留问题
- 下次改进方向
</tucao>
```

**吐槽内容示例**：
```
<tucao>
好了好了，Master又来活了。看看这次的输入...哦，要写战斗场景啊。
那得注意：
1. 节奏要快，短句为主
2. 不能用那些老掉牙的比喻
3. 别让角色突然失去战斗力
得选个合适的文风，用那个「节奏大师」文风吧，适合打戏。
</tucao>
```

#### 优缺点分析

**优点**：
- ✅ **注意力集中**：比传统思维链更有效，不会"写一半忘了"
- ✅ **指令放大**：一句话指令经过吐槽放大后效果显著
- ✅ **实时纠错**：在创作过程中动态调整
- ✅ **人格化**：增加交互趣味性
- ✅ **模块化**：可以轻松添加新的检查点

**缺点**：
- ❌ **Token消耗大**：每次输出至少4次吐槽
- ❌ **格式要求严格**：需要正确闭合标签
- ❌ **可能影响沉浸感**：破坏第四面墙

#### 适用场景
- 长篇故事创作
- 需要严格遵守复杂规则的场景
- 多轮对话中保持一致性
- 角色扮演中需要元认知

#### 推荐度：⭐⭐⭐⭐⭐

**代码示例（简化版）**：
```yaml
- system: |
    # 吐槽式创作流程
    <tucao>首次吐槽：理解任务并规划</tucao>
    正文第一段
    <tucao>中间吐槽：检查是否符合规则</tucao>
    正文第二段
    <tucao>最终吐槽：总结完成度</tucao>
```

### 2. 思维链/CoT机制 ⭐⭐⭐⭐⭐

#### 功能描述
在正文生成前进行结构化思考，确保输出质量和逻辑一致性。

#### 实现方式对比

**A. 传统思维链（Sudachi/Aether）**
```yaml
<thinking>
- 当前场景分析
- 角色状态
- 后续情节规划
- 文风确认
- 格式检查
</thinking>
### 正文
```

**B. 中文思维链（Acheron）**
```yaml
<CoT>
# CoT: 文风名称
【阶段一：理解与分析】
- Step 1: 场景定位
- Step 2: 核心矛盾
- Step 3: 角色动机

【阶段二：创作规划】
- Step 1: 情节节拍
- Step 2: 文风选择
- Step 3: 细节安排

【约束复核】
- 是否满足所有要求
</CoT>
```

**C. 吐槽式思维链（Izumi）**
```yaml
<thinking>
好了，又来活了。这次Master想要什么呢？
- 先看看当前情况...
- 得注意文风选择
- 字数要求是多少来着
- 别忘了那个禁用词列表

行，开始写！
</thinking>
```

#### 优缺点分析

**优点**：
- ✅ **逻辑保证**：系统化思考减少错误
- ✅ **一致性**：多轮对话保持连贯
- ✅ **可调试**：可以查看AI的思考过程
- ✅ **灵活性**：可以为不同文风定制CoT

**缺点**：
- ❌ **Token开销**：增加30-50%的token消耗
- ❌ **速度影响**：生成时间增加
- ❌ **可能过度思考**：有时反而降低自然度

#### 适用场景
- 复杂剧情创作
- 多角色互动
- 需要严格遵守规则
- 技术性写作

#### 推荐度：⭐⭐⭐⭐⭐

**实用技巧**：
```yaml
# 简化版思维链（推荐）
<thinking>
1. 场景：${简要描述}
2. 目标：${本次要达成什么}
3. 风格：${文风选择}
4. 注意：${关键禁忌}
</thinking>
```

### 3. 变量系统 ⭐⭐⭐⭐

#### 功能描述
使用SillyTavern的变量机制实现动态配置和功能模块化。

#### 实现方式

**A. 变量初始化（Sudachi）**
```yaml
# 变量声明
{{setvar::style::}}
{{setvar::cotdlc::}}
{{setvar::summary1::}}
{{setvar::antisimile::}}
# ... 50+ 变量
```

**B. 变量赋值**
```yaml
# 设置文风
{{setvar::style::
  文风: 日式轻小说
  特点:
    - 对话占比大
    - 简洁轻盈
    - 人设鲜明
}}

# 设置功能开关
{{setvar::summary1::<summary>}}
{{setvar::summary2::</summary>}}
```

**C. 变量使用**
```yaml
# 在输出格式中使用
{{getvar::summary1}}
${正文内容}
{{getvar::summary2}}
```

#### 核心应用场景

**1. 功能开关**
```yaml
# 初始化为空（关闭）
{{setvar::summary::}}

# 需要时赋值（开启）
{{setvar::summary::
  <summary>
  ${摘要内容}
  </summary>
}}
```

**2. 模块化配置**
```yaml
# 文风库
{{setvar::style_ln::轻小说文风配置}}
{{setvar::style_gal::Galgame文风配置}}
{{setvar::style_wuxia::武侠文风配置}}

# 根据场景选择
当前文风：{{getvar::style_ln}}
```

**3. 条件逻辑**
```yaml
# 设置条件变量
{{setvar::is_nsfw::当前是否为性场景？}}

# 在CoT中使用
{{getvar::is_nsfw}}
如果是，使用情色文风；否则跳过
```

#### 优缺点分析

**优点**：
- ✅ **模块化**：功能可以独立开关
- ✅ **可维护性**：修改配置无需改动主逻辑
- ✅ **复用性**：可以轻松组合不同模块
- ✅ **用户友好**：用户可以自定义配置

**缺点**：
- ❌ **依赖SillyTavern**：不通用
- ❌ **调试困难**：变量多了容易混乱
- ❌ **性能影响**：大量变量替换有开销

#### 适用场景
- 多功能预设
- 需要用户自定义配置
- 模块化设计
- 条件渲染

#### 推荐度：⭐⭐⭐⭐

**最佳实践**：
```yaml
# 1. 分类变量
# 文风类
{{setvar::style::}}
{{setvar::style2::}}

# 功能类
{{setvar::cot::}}
{{setvar::summary::}}

# 规则类
{{setvar::antisimile::}}
{{setvar::antiprocess::}}

# 2. 使用命名约定
{{setvar::enable_summary::}}  # enable_ 前缀表示开关
{{setvar::config_wordcount::}} # config_ 前缀表示配置
{{setvar::text_summary::}}     # text_ 前缀表示文本内容
```

### 4. 反八股机制 ⭐⭐⭐⭐⭐

#### 功能描述
通过禁用词列表、写作模拟、反向思考等技术，避免AI生成陈词滥调的"八股文"。

#### 实现方式

**A. 禁用词列表（直接禁止）**
```yaml
避免使用:
  四字成语（无意义辞藻堆砌）
  指节发白、指节如何（聚焦于无关细微细节）
  声音、语气类描写（陈词滥调）
  不易察觉的（打乱节奏，无法感知）
  不是…而是…转折句式（无关联想）

禁止比喻:
  - 如鹰隼般锐利
  - 稳如磐石
  - 心如刀绞
  - 如遭雷击
```

**B. writing_antThinking（Kemini/Aether）**
```yaml
# 每段前模拟两种写法
<!-- writing_antThinking:
第一类（八股）：
"她的话像一颗石子，砸入平静的湖面，荡起涟漪"

第二类（反八股）：
"她说了那句话。他愣了一下。"
-->

# 正文必须使用第二类
她说了那句话。他愣了一下。
```

**C. 情绪反八股（emotion_antThinking）**
```yaml
<!-- emotion_antThinking:
第一类（过度情绪）：角色崩溃大哭
第二类（克制情绪）：角色眼眶微红，转过身去
-->

# 正文使用第二类
她眼眶微红，转过身去。
```

**D. 对话反八股（dialogue_antThinking）**
```yaml
<!-- dialogue_antThinking:
模拟要求风格: 正式、文学化
模拟内容: "我对此深感遗憾"
正文要求风格: 生活化、情绪化
-->

# 正文
"抱歉啊...我真的..."
```

#### 多层次反八股策略

**1. 句式层面**
```yaml
禁止:
  - 连续使用"主语+动词"
  - 过度使用形容词副词
  - 固定的修辞模式

推荐:
  - 短句和长句交错
  - 以动作/环境开头
  - 省略主语
```

**2. 修辞层面**
```yaml
禁止:
  - 陈词滥调的比喻
  - "像...一样"的滥用
  - 拟人化描写

推荐:
  - 原创、贴切的意象
  - 直接描述
  - 具体的感官细节
```

**3. 情感层面**
```yaml
禁止:
  - "如遭雷击"
  - "心如刀绞"
  - "大脑一片空白"

推荐:
  - 具体的生理反应
  - 动作暗示情绪
  - 克制的表达
```

**4. 视角层面**
```yaml
禁止:
  - 眼神深邃
  - 眼中闪过复杂情绪
  - 目光如炬

推荐:
  - 只描述视线方向
  - 物理状态（眯眼、瞪眼）
  - 不解读含义
```

#### 优缺点分析

**优点**：
- ✅ **显著提升质量**：避免千篇一律
- ✅ **增加真实感**：更贴近实际写作
- ✅ **可定制**：可以针对性添加禁用词
- ✅ **多层防护**：从多个维度避免八股

**缺点**：
- ❌ **可能过度**：限制太多反而僵硬
- ❌ **理解成本**：需要AI充分理解规则
- ❌ **效果不稳定**：强模型效果好，弱模型可能失效

#### 适用场景
- 文学创作
- 追求个性化风格
- 避免AI味
- 高质量对话生成

#### 推荐度：⭐⭐⭐⭐⭐

**实用示例**：
```yaml
# 最小化反八股配置
<anti_bagu>
禁用词:
  比喻类: [像...一样, 仿佛, 如同]
  情绪类: [如遭雷击, 心如刀绞, 崩溃]
  描写类: [不易察觉, 微妙, 复杂的情绪]

替代方案:
  - 用具体动作代替抽象情感
  - 用白描代替修辞
  - 用短句代替长句
</anti_bagu>
```

### 5. 文风控制机制 ⭐⭐⭐⭐⭐

#### 功能描述
通过详细的文风定义和示例，精确控制AI的写作风格。

#### 实现方式

**A. 文风模板库（Sudachi/Izumi）**

5个预设共提供了30+种文风模板：

```yaml
# 轻小说风格
文风: 日式轻小说
特点:
  - 大量对话
  - 内心独白
  - 简洁语言
  - 青少年受众

# 武侠风格
文风: 古龙式武侠
特点:
  - 短句断句
  - 节奏感强
  - 意境渲染
  - 冷峻白描

# 情色风格
文风: H小说/官能小说
特点:
  - 五感描写
  - 淫语使用
  - 节奏控制
  - 禁用词规范
```

**B. 详细文风定义（acheron）**

```yaml
<writing_style>
文风: 海明威式写实
1. 风格锚定:
   - 模仿海明威、卡佛
   - 冰山理论

2. 细节捕获:
   - 精准刻画物理细节
   - 用细节暗示性格

3. 行为优先级:
   - P1: 外部表现
   - P2: 心理活动

4. 语言质感:
   - 准确、明晰、克制
   - 避免过度修饰

5. 禁止项:
   - 总结性陈述
   - 直白心理
   - 过多比喻
</writing_style>
```

**C. 文风组合（Izumi）**

```yaml
# 春秋笔法·泉此方主观叙事体
narrative_identity:
  observer_stance: "ACG文化背景的宅女作家"
  core_method: "通过选择细节暗示态度"

pov_control:
  third_person_with_bias:
    - 关注ACG相关细节
    - 游戏/动漫术语
    - 对套路的熟悉感

  seamless_first_person:
    triggers: [强烈情绪, 关键决策]
    transition: 自然过渡
    permissions: [内心独白, 感官体验]
```

#### 文风分类体系

**1. 按媒介分类**
- 小说：轻小说、网文、纯文学
- 剧本：舞台剧、影视剧
- 互动：Galgame、TRPG

**2. 按风格分类**
- 写实：海明威、阿城、鲁迅
- 意识流：伍尔芙、普鲁斯特
- 新派：村上春树、东野圭吾
- 古典：金庸、古龙

**3. 按功能分类**
- 叙事向：推动剧情
- 氛围向：营造情绪
- 角色向：塑造人物
- 互动向：玩家代入

#### 优缺点分析

**优点**：
- ✅ **高度可控**：精确控制输出风格
- ✅ **丰富多样**：30+种文风可选
- ✅ **易于切换**：变量控制即可
- ✅ **质量稳定**：详细规范保证一致性

**缺点**：
- ❌ **学习成本**：需要理解各种文风特点
- ❌ **配置复杂**：详细定义token消耗大
- ❌ **可能冲突**：多个文风组合可能矛盾

#### 适用场景
- 文学创作
- 角色扮演
- 故事生成
- 风格迁移

#### 推荐度：⭐⭐⭐⭐⭐

**实用模板**：
```yaml
# 简化文风定义
<writing_style>
文风名称: ${名称}

核心特征:
  - ${特征1}
  - ${特征2}
  - ${特征3}

必须做:
  - ${要求1}
  - ${要求2}

禁止做:
  - ${禁止1}
  - ${禁止2}

示例:
${具体示例}
</writing_style>
```

### 6. 对话格式规范 ⭐⭐⭐⭐

#### 功能描述
统一对话的格式规范，包括引号使用、旁白标记、动作描述等。

#### 实现方式

**A. 基础格式（通用）**
```yaml
[user input rule]
1. 双引号内为发言
2. 星号内为内心想法
3. 其余内容为旁白
4. 内心想法和旁白无法被读取
```

**B. Galgame格式（Sudachi）**
```yaml
强制格式:
【用户】："对话内容"
【旁白】：客观动作描写（不代替内心）
【角色】："对话内容"

正确示例:
【旁白】：指尖轻颤着抓住衣角
【角色】："呜...请别这样看..."

错误示例:
【旁白】：她感到非常害羞（代心理）
```

**C. 自由引语（crychic）**
```yaml
<discourse_mode>
- 省掉所有引导词
- 抹去叙述者声音
- 人物自己说话
- 无需特殊符号包裹
</discourse_mode>

示例:
错误：他说："我不同意。"
正确：我不同意。（直接呈现）
```

**D. ASMR格式（仅声音）**
```yaml
禁止:
- 视觉描写
- 心理活动
- 叙述旁白
- 括号标注

只能有:
- 对话台词
- 拟声词（融入台词）
- 语气词

示例:
"嗯...啊...♡轻一点...听到水声了吗...咕啾♡"
```

#### 多样化格式方案

**1. 简洁格式**
```
"对话"
动作。
"对话"
```

**2. 详细格式**
```
角色A："对话"动作描写。
角色B：*内心想法*
旁白描写。
角色B："回应"
```

**3. 剧本格式**
```
（动作/场景描写）
角色：「对话」
```

#### 优缺点分析

**优点**：
- ✅ **清晰易读**：格式统一便于阅读
- ✅ **角色区分**：明确谁在说话
- ✅ **信息分层**：对话、动作、内心分离
- ✅ **适配场景**：不同格式适合不同应用

**缺点**：
- ❌ **格式限制**：可能降低自然度
- ❌ **遵守困难**：AI容易打破格式
- ❌ **平台依赖**：某些格式需要特定渲染

#### 适用场景
- 角色扮演
- 视觉小说
- 语音生成
- 剧本创作

#### 推荐度：⭐⭐⭐⭐

### 7. 摘要机制 ⭐⭐⭐⭐

#### 功能描述
在对话后生成简洁摘要，便于长对话的上下文管理。

#### 实现方式

**A. 基础摘要（Kemini/crychic）**
```yaml
每次正文结束后生成摘要:
<summary>
用100字概括本次剧情
- 客观陈述
- 不做评价
- 记录事实

时间：YYYY-MM-DD HH:MM
地点：具体地点
</summary>
```

**B. 摘要正则控制（Izumi）**
```yaml
# 10楼以下：不发送摘要
# 10楼以上：只发送摘要

<details>
<summary>摘要</summary>
${摘要内容}
</details>
```

**C. 结构化摘要（acheron）**
```yaml
<summary>
When: 时间（YYYY-MM-DD HH:MM）
Who: 涉及角色
Where: 地点 + 环境特征
What: [事件1] → [过程] → [结果]
Feel: 具体动作/神态（禁止抽象词）
Key: 关键细节

【已发生】
</summary>
```

#### 摘要最佳实践

**1. 内容要求**
```yaml
必须包含:
  - 时间（精确到分钟）
  - 地点（三级定位）
  - 人物（全名）
  - 事件链（因果关系）
  - 关键对话

禁止包含:
  - 主观评价
  - 情感推测
  - 升华总结
  - 模糊表达
```

**2. 格式规范**
```yaml
# 简洁版（100字）
<summary>
${核心事件}
时间：${具体时间}
地点：${具体地点}
</summary>

# 详细版（200字）
<summary>
【事件】${发生了什么}
【人物】${谁参与了}
【结果】${导致了什么}
【时间】${YYYY-MM-DD HH:MM~HH:MM}
【地点】${大/中/小地点}
</summary>
```

#### 优缺点分析

**优点**：
- ✅ **上下文压缩**：节省token
- ✅ **快速回顾**：便于续写
- ✅ **记忆增强**：长对话不遗忘
- ✅ **结构化**：便于检索和分析

**缺点**：
- ❌ **额外开销**：每次都要生成
- ❌ **可能失真**：摘要可能丢失细节
- ❌ **格式要求**：需要正确闭合标签

#### 适用场景
- 长篇对话
- 多轮剧情
- 需要回顾的场景
- 上下文管理

#### 推荐度：⭐⭐⭐⭐

**实用配置**：
```yaml
# 可折叠摘要（推荐）
<details>
<summary>第${N}轮摘要</summary>

【核心事件】${简述}
【时间地点】${时间} @ ${地点}
【关键对话】"${重要对话}"
【情节推进】${下一步}
```

- ❌ **可能晦涩**：过度使用可能影响可读性
- ❌ **AI理解**：不是所有模型都能准确把握

#### 适用场景

- 文学性创作
- 跨文化故事
